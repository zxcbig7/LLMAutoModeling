2框架核心觀念

動手前先建立一張地圖。OptimFoundation 把 CPLEX 包成幾個你只需要會用的積木。

2.1 一個最佳化問題長什麼樣

任何線性最佳化問題都由四個東西組成。先用直覺理解：

元素白話框架對應
Sets 集合有哪些東西要決策（哪些員工、哪些日期、哪些班別）List<string> / List<DateTime>
Parameters 參數已知的固定數字（每日需求人數、罰分權重、跨組別成本）Parameter_Xxx : ParameterBase
Decision Variables 決策變數要求解器幫你決定的未知數（某人某天排哪一班）VariableB/X_Xxx : VariableBase
Objective 目標式要最小化或最大化的東西（總罰分）CreateMinimize() / CreateMaximize()
Constraints 限制式必須滿足的規則（每天恰排一班、需求要被滿足）CreateEqual() / CreateLessEqual() 等

2.2 三種決策變數 = MILP 的「MI」

MILP（Mixed-Integer Linear Programming）的「Mixed-Integer」就是指變數可以混用連續與整數。框架用類別名稱前綴區分，並用不同的 Build*Vs 建立：

前綴型別建立方法本模型用途
VariableB_Binary 二元（0 / 1）BuildBVs<T>()排班決策 + 違規指示變數
VariableX_Continuous 連續（≥ 0）BuildCVs<T>()休假不足平均 / 週末休不足的「差距量」
VariableI_Integer 整數（≥ 0）BuildIVs<T>()（本模型未使用，框架仍支援）
本模型的變數結構

核心決策只有一個：VariableB_ShiftAssign[Date, Employee, Group]——某員工某天是否排某班別（含休假 O）。其餘的二元變數都是指示變數（indicator）：用來「偵測」某種不希望的情況有沒有發生（例如連上六天、夜接早），再到目標式被罰分。兩個連續變數則記錄「差了多少」（休假比平均少幾天）。

2.3 Pool 機制：AddLHS / AddRHS / CreateXxx

這是整個框架最核心、也最容易誤解的部分。你不直接寫出 3x + 2y ≤ 10 這種式子，而是把一條限制式「分次累加」進一個暫存區（Pool），最後呼叫 Create* 收尾：

Pool 機制示意csharp

// 目標：某員工某天排班數總和 = 1（每天恰一個班別）

foreach (var g in dataload.Group)
    engine.AddLHS(1.0, new VariableB_ShiftAssign { Date = d, Employee = e, Group = g });
engine.AddRHS(1);                                  // 右側常數

engine.CreateEqual($"Constraint_OneGroup@{d:yyyy_MM_dd}@{e}");  // 收尾：建立 = 限制式並清空 Pool

運作規則有三條，記住就不會出錯：

  1. 累加：每次 AddLHS / AddRHS 只是往 Pool 丟一項，可以丟很多次（常用在 foreach 迴圈裡逐項加總）。
  2. 收尾即清空：呼叫 CreateLessEqual / CreateGreatEqual / CreateEqual 後，Pool 自動清空，下一條限制式重新開始。
  3. 方向不可猜：數學模型寫 ≤ 就用 CreateLessEqual、≥ 用 CreateGreatEqual、= 用 CreateEqual，禁止翻轉。

框架在 Create* 時其實是計算 LHS − RHS 後再跟常數比較，所以你可以把變數放在右側（移項），這在「指示變數連動」型限制式很好用，稍後每一條都會用到。

2.4 OptModel：組裝模型的唯一進入點

你不直接管理引擎生死。框架的 OptModel 是一個註冊式 pipeline：用鏈式方法「登記」設定與各建構步驟，呼叫 Execute() 後依序把它們叫起來、求解、釋放資源。

new OptModel("專案名") │ ▼ .UseConfig(() => new CplexConfig{...}) 套用求解設定 │ ▼ .AddVariables(e => ...) 登記建變數步驟（先） │ ▼ .AddModel(e => ...) 登記建目標式 + 限制式（後） │ ▼ .OnSolved(e => ...) 登記求解成功後的輸出 │ ▼ .Execute() ──▶ bool 建模 → 求解 → OnSolved → Dispose true = 找到 Optimal 或 Feasible 解
底層仍是 OptEngine

每個 .AddVariables / .AddModel 的 lambda 參數 e 就是底層 OptEngine——你在裡面呼叫 BuildBVs / AddLHS / CreateLessEqual 等建模窗口。OptModel 只負責「在對的時機、用對的順序」驅動這顆引擎，保證 build / solve / dispose 不漏。

3問題情境：護理人員月排班

我們要排一張單月（31 天）的護理班表。每天每個班別都有固定的人力需求一定要補滿；同時要盡量避免一堆「不健康」的排法。這是一個典型的 Nurse Rostering（護理排班）問題，天生適合用「硬約束 + 軟性懲罰」的 MILP 來解。

情境

某病房有 16 位護理師，每天要把每個人排進五種狀態之一：休(O)、早班(D)、午班(E)、夜班(N)、行政(C)。每天各工作班別的人數需求一定要剛好補滿。在此前提下，醫院希望盡量避免：連上六天班、夜班接早班這類不良順序、做一休一做、跨自己主責的組別支援、某人休假比同儕少太多、週末休得太少…等。每種情況各有一個罰分權重，目標是讓總罰分最低。

兩種建立實例的寫法

類別預設只宣告屬性（有隱含的無參數建構子），平常用 object initializer 建立。若需要位置式 / 動態建構（逐列讀 CSV、用迴圈帶 object[] 進來），就在類別內加一個 params object[] 建構子呼叫 InitClassBySets——它會依屬性宣告順序逐一填值並自動轉型。

兩種寫法csharp

// 寫法 A（預設）：object initializer —— 欄位明確、可讀性最高

var p = new Parameter_ShiftDemand { Date = d, Group = "D", QTY = 5 };

// 寫法 B（位置式 / 動態）：在類別內加一個 params object[] 建構子

public class Parameter_ShiftDemand : ParameterBase
{
    public DateTime Date  { get; set; }
    public string   Group { get; set; } = string.Empty;
    public double   QTY   { get; set; }

    public Parameter_ShiftDemand(params object[] sets) => InitClassBySets(sets);
}
// 之後依「屬性宣告順序」位置式帶入（型別自動轉換）：

var p2 = new Parameter_ShiftDemand(d, "D", 5);
用寫法 B 的兩個前提

① 引數順序與數量必須與屬性宣告完全一致（Date → Group → QTY），否則 InitClassBySets 會丟例外。② 一旦宣告了這個建構子，隱含的無參數建構子就消失——若該類別同時還要用 object initializer 或被 CsvCtrl.BuildParameter<T>（需 new()）載入，就再補一個無參數建構子。同一機制對所有 Variable 類別一樣適用。

二維 / 多鍵參數怎麼存

像 這種「矩陣」型參數，框架的做法是攤平成一筆一筆的 row：每個 (Date, Group) 組合一筆 Parameter_ShiftDemand。讀取時用 LINQ 依索引找出對應的那筆（FirstOrDefault(x => x.Date == d && x.Group == g)）。

5.3 Dataload：填入資料、衍生 Sets、放罰分權重

這是整個專案唯一存放「實際數字」的地方。它負責：產生 Sets（員工、班別、日期）、填參數資料（需求、跨組別成本、不良轉換、預排班）、放純量罰分權重，以及求解後的結果輸出。

Set/Dataload.cscsharp

using OptimFoundation.Cplex;
using OptimFoundation.Core;
using HospitalRostering.Parameter;
using HospitalRostering.Variable;

namespace HospitalRostering.Set
{
    public class Dataload
    {
        // ── 罰分權重（純量參數，禁止寫死在 Constraint/Objective） ──────

        public double Penalty_OffOneDay     = 0.1;   // w1 做一休一做

        public double Penalty_DoubleOffLT2  = 0.1;   // w2 整月無連休 2 天

        public double Penalty_GroupMismatch = 0.2;   // w3 跨組別支援

        public double Penalty_NightToDay    = 0.2;   // w4 不良班別轉換

        public double Penalty_BelowAVG      = 0.1;   // w5 休假低於平均

        public double Penalty_SixDay        = 1.0;   // w6 連續工作 6 天

        public double Penalty_Weekend4Day   = 0.1;   // w7 週末休不足 4 天

        public double Penalty_Backup        = 0.0;   // Backup 支援成本


        // ── Sets ──────────────────────────────────────────────────────

        public List<string>   Employee = new();
        public List<string>   Group    = new();
        public List<DateTime> Date     = new();

        // ── Parameters ────────────────────────────────────────────────

        public List<Parameter_ShiftDemand> parameter_ShiftDemand = new();
        public List<Parameter_NightToDay>  parameter_NightToDay  = new();
        public List<Parameter_CrossGroup>  parameter_CrossGroup  = new();
        public List<Parameter_PreAssign>   parameter_PreAssign   = new();
        public List<Parameter_BackupGroup> parameter_BackupGroup = new();

        // 班別群組（O 必須在內，代表休假）

        enum GroupE { O, D, E, N, C }

        public Dataload()
        {
            // ① 班別

            Group.AddRange(Enum.GetNames(typeof(GroupE)));

            // ② 員工：依主責班別人數展開 E1..E16

            (int Count, string Main)[] empByGroup = [(7, "D"), (5, "E"), (3, "N"), (1, "C")];
            int totalEmp = empByGroup.Sum(x => x.Count);
            for (int e = 1; e <= totalEmp; e++) Employee.Add($"E{e}");

            // ③ 跨組別成本：員工上「非主責、非 O」的班別都扣 Penalty_GroupMismatch

            int idx = 1;
            foreach (var grp in empByGroup)
            {
                var crossGroups = Group.Where(g => g != grp.Main && g != "O").ToList();
                for (int k = 0; k < grp.Count; k++)
                {
                    foreach (var g in crossGroups)
                        parameter_CrossGroup.Add(new Parameter_CrossGroup
                        { Employee = $"E{idx}", Group = g, QTY = Penalty_GroupMismatch });
                    idx++;
                }
            }

            // ④ Backup 例外：E1 支援 C 不扣分 → 覆寫該筆成本

            parameter_BackupGroup.Add(new Parameter_BackupGroup { Employee = "E1", Group = "C" });
            foreach (var b in parameter_BackupGroup)
            {
                var hit = parameter_CrossGroup.FirstOrDefault(x => x.Employee == b.Employee && x.Group == b.Group);
                if (hit != null) hit.QTY = Penalty_Backup;
            }

            // ⑤ 不良班別轉換對 (昨天 → 今天)

            void Bad(string pre, string cur) =>
                parameter_NightToDay.Add(new Parameter_NightToDay
                { PreGroup = pre, Group = cur, QTY = Penalty_NightToDay });
            Bad("N", "D"); Bad("N", "E"); Bad("N", "C");          // 夜 → 早/午/行

            Bad("E", "D"); Bad("E", "C"); Bad("E", "N");          // 午 → 早/行/夜

            Bad("D", "N"); Bad("C", "N");                          // 早/行 → 夜


            // ⑥ 排程月份（2026-01）與每日需求

            int year = 2026, month = 1;
            int days = DateTime.DaysInMonth(year, month);
            var rng = new Random(20260101);   // 固定種子，需求可重現

            for (int day = 1; day <= days; day++)
            {
                var d = new DateTime(year, month, day);
                Date.Add(d);
                parameter_ShiftDemand.Add(new() { Date = d, Group = "D", QTY = rng.Next(4, 6) }); // 4~5

                parameter_ShiftDemand.Add(new() { Date = d, Group = "E", QTY = 3 });
                parameter_ShiftDemand.Add(new() { Date = d, Group = "N", QTY = 2 });
                parameter_ShiftDemand.Add(new() { Date = d, Group = "C", QTY = 1 });
            }

            // ⑦ 預排班（硬性鎖定）

            parameter_PreAssign.Add(new() { Date = new(2026, 1, 1), Employee = "E1", Group = "E" });
            parameter_PreAssign.Add(new() { Date = new(2026, 1, 1), Employee = "E3", Group = "O" });
            parameter_PreAssign.Add(new() { Date = new(2026, 1, 2), Employee = "E2", Group = "D" });
            parameter_PreAssign.Add(new() { Date = new(2026, 1, 2), Employee = "E3", Group = "E" });
        }

        // ── 結果輸出（求解成功後呼叫） ──────────────────────────────────

        public void WriteResult(OptEngine engine)
        {
            Logging.Info("═══════════════ 月班表（每列一位員工） ═══════════════");

            var assign = engine.GetSetVarValues<VariableB_ShiftAssign>();   // key: 類別@Date@Emp@Group

            foreach (var e in Employee)
            {
                var row = Date.Select(d =>
                {
                    var g = Group.FirstOrDefault(gg =>
                        assign.TryGetValue($"VariableB_ShiftAssign@{d:yyyy-MM-dd}@{e}@{gg}", out var v) && v > 0.5);
                    return g ?? "?";
                });
                Logging.Info($"  {e,-4}: {string.Join(" ", row)}");
            }

            Logging.Info($"  最小總罰分 = {engine.GetObjectiveValue():F2}");
            Logging.Info($"  MIP Gap   = {engine.MIPGap:P2}");
            Logging.Info("══════════════════════════════════════════════════════");

            // 匯出 CSV（Solution/VariableB_ShiftAssign.csv）

            FolderDir.Solution.CreateFolder();
            CsvCtrl.SaveSolutionToCSV<VariableB_ShiftAssign>(engine, "HospitalRostering", "USER");

using OptimFoundation.Core;
using HospitalRostering.Set;

5.5 Objective：加權懲罰目標式

目標式也是用 Pool：把每一個違規指示變數乘上對應罰分 AddLHS 進去，最後 CreateMinimize()。權重一律從 dataload 查，禁止寫死。沒有任何 RHS——目標式只是一條要被最小化的線性式。

Objective/ObjectiveFunction.cscsharp

using OptimFoundation.Cplex;
using OptimFoundation.Core;
using HospitalRostering.Set;
using HospitalRostering.Variable;

namespace HospitalRostering.Objective
{
    /// <summary>min  Σ(w1·s^off1 + w6·s^six + w3·s^mis + w4·s^ntd) + Σ(w2·s^dlt + w5·z^avg + w7·z^wkd)</summary>

    public class ObjectiveFunction
    {
        private OptEngine optEngine;
        private Dataload  dataload;

        public ObjectiveFunction(Dataload dataload, OptEngine engine)
        {
            this.optEngine = engine;
            this.dataload  = dataload;
        }

        public void Build()
        {
            // ① 每日違規（Date × Employee）

            dataload.Date.ForEach(d =>
                dataload.Employee.ForEach(e =>
                {
                    optEngine.AddLHS(dataload.Penalty_OffOneDay,     new VariableB_Off1Day      { Date = d, Employee = e });
                    optEngine.AddLHS(dataload.Penalty_SixDay,        new VariableB_SixDayWork   { Date = d, Employee = e });
                    optEngine.AddLHS(dataload.Penalty_GroupMismatch, new VariableB_GroupMismatch{ Date = d, Employee = e });
                    optEngine.AddLHS(dataload.Penalty_NightToDay,    new VariableB_NightToDay   { Date = d, Employee = e });
                }));

            // ② 每人違規 / 差距量（Employee）

            dataload.Employee.ForEach(e =>
            {
                optEngine.AddLHS(dataload.Penalty_DoubleOffLT2, new VariableB_DoubleOffLT2 { Employee = e });
                optEngine.AddLHS(dataload.Penalty_BelowAVG,     new VariableX_BelowAVG     { Employee = e });
                optEngine.AddLHS(dataload.Penalty_Weekend4Day,  new VariableX_WeekendLT4   { Employee = e });
            });

            optEngine.CreateMinimize();
            Logging.Info("目標式：min 加權違規總和");
        }
    }
}

5.6 Constraints：每條限制式各一個類別

每條限制式獨立成一個 Constraint_Xxx 類別、繼承 ConstraintBase。ConstraintName 自動取得類別名稱，ConstraintCount 計數。限制式名稱務必帶上迴圈索引（日期、員工、規則…），否則框架會把同名限制式當重複而略過。

C1 — 每人每天恰一個班別（Balance，=）

Constraint/Constraint_OneGroup.cscsharp

using OptimFoundation.Cplex;
using OptimFoundation.Core;
using HospitalRostering.Set;
using HospitalRostering.Variable;

namespace HospitalRostering.Constraint
{
    /// <summary>[C1] Σ_g y[e,d,g] = 1   ∀ e, d</summary>

    public class Constraint_OneGroup : ConstraintBase
    {
        private OptEngine optEngine;
        private Dataload  dataload;

        public Constraint_OneGroup(Dataload dataload, OptEngine engine)
        { this.optEngine = engine; this.dataload = dataload; }

        public void Build()
        {
            dataload.Date.ForEach(d =>
                dataload.Employee.ForEach(e =>
                {
                    dataload.Group.ForEach(g =>
                        optEngine.AddLHS(1, new VariableB_ShiftAssign { Date = d, Employee = e, Group = g }));
                    optEngine.AddRHS(1);
                    optEngine.CreateEqual($"{ConstraintName}@{d:yyyy_MM_dd}@{e}");
                    ConstraintCount++;
                }));
            Logging.Info($"[{ConstraintName}] {ConstraintCount}");
        }
    }
}

C4 — 連續工作 6 天指示（滑動視窗 + 上下界）

這是最能展示「指示變數雙向鎖定」的一條。上界：視窗內任一天休假就壓住 ；下界：整個視窗都沒休就頂起 。視窗不足 6 天（月初）直接跳過。各條名稱都帶索引確保唯一。

Constraint/Constraint_SixDayWork.cscsharp

using OptimFoundation.Cplex;
using OptimFoundation.Core;
using HospitalRostering.Set;
using HospitalRostering.Variable;

namespace HospitalRostering.Constraint
{
    /// <summary>[C4] s^six = 1 ⇔ 視窗 6 天全勤（無休）</summary>

    public class Constraint_SixDayWork : ConstraintBase
    {
        private OptEngine optEngine;
        private Dataload  dataload;

        public Constraint_SixDayWork(Dataload dataload, OptEngine engine)
        { this.optEngine = engine; this.dataload = dataload; }

        public void Build()
        {
            int duration = 6;
            dataload.Date.ForEach(d =>
                dataload.Employee.ForEach(e =>
                {
                    var window = dataload.Date.Where(sd => d.AddDays(-duration) < sd && sd <= d).ToList();
                    if (window.Count < duration) return;   // 月初視窗不足，跳過


                    // 上界：s^six ≤ 1 - y[e,τ,O]   （每天一條，名稱帶 τ）

                    window.ForEach(sd =>
                    {
                        optEngine.AddLHS(1, new VariableB_SixDayWork  { Date = d, Employee = e });
                        optEngine.AddRHS(1);
                        optEngine.AddRHS(-1, new VariableB_ShiftAssign { Date = sd, Employee = e, Group = "O" });
                        optEngine.CreateLessEqual($"{ConstraintName}_ub@{d:yyyy_MM_dd}@{e}@{sd:MM_dd}");
                        ConstraintCount++;
                    });

                    // 下界：s^six ≥ 1 - Σ_τ y[e,τ,O]

                    optEngine.AddLHS(1, new VariableB_SixDayWork { Date = d, Employee = e });
                    optEngine.AddRHS(1);
                    window.ForEach(sd =>
                        optEngine.AddRHS(-1, new VariableB_ShiftAssign { Date = sd, Employee = e, Group = "O" }));
                    optEngine.CreateGreatEqual($"{ConstraintName}_lb@{d:yyyy_MM_dd}@{e}");
                    ConstraintCount++;
                }));
            Logging.Info($"[{ConstraintName}] {ConstraintCount}");
        }
    }

C8 + C9 — 連休旗標與「整月至少連休一次」（同一類別兩段）

這條把「偵測連休 2 天的旗標 」和「整月加總要求 」放在同一個 Constraint。視窗 2 天或 3 天分別處理（月初退化）。

Constraint/Constraint_DoubleOffLT2.cscsharp

using OptimFoundation.Cplex;
using OptimFoundation.Core;
using HospitalRostering.Set;
using HospitalRostering.Variable;

namespace HospitalRostering.Constraint
{
    /// <summary>[C8] 連休旗標 s^dfl ；[C9] Σ_d s^dfl + 2·s^dlt ≥ 2</summary>

    public class Constraint_DoubleOffLT2 : ConstraintBase
    {
        private OptEngine optEngine;
        private Dataload  dataload;

        public Constraint_DoubleOffLT2(Dataload dataload, OptEngine engine)
        { this.optEngine = engine; this.dataload = dataload; }

        public void Build()
        {
            int duration = 3;

            // ── C8：每天的連休旗標 ───────────────────────────────────

            dataload.Date.ForEach(d =>
            {
                var window = dataload.Date.Where(sd => d.AddDays(-duration) < sd && sd <= d).ToList();
                if (window.Count < 2) return;

                dataload.Employee.ForEach(e =>
                {
                    var preD = d.AddDays(-1);
                    optEngine.AddLHS(1, new VariableB_DoubleOffFlag { Date = d, Employee = e });

                    if (window.Count == 2)
                    {
                        // s^dfl ≥ y[d,O] + y[d-1,O] - 1

                        optEngine.AddRHS(1, new VariableB_ShiftAssign { Date = d,    Employee = e, Group = "O" });
                        optEngine.AddRHS(1, new VariableB_ShiftAssign { Date = preD, Employee = e, Group = "O" });
                        optEngine.AddRHS(-(2 - 1));
                    }
                    else // window.Count == 3

                    {
                        var prepreD = d.AddDays(-2);
                        // s^dfl ≥ y[d,O] + y[d-1,O] + (1 - y[d-2,O]) - 2

                        optEngine.AddRHS(1, new VariableB_ShiftAssign { Date = d,    Employee = e, Group = "O" });
                        optEngine.AddRHS(1, new VariableB_ShiftAssign { Date = preD, Employee = e, Group = "O" });
                        optEngine.AddRHS(1);
                        optEngine.AddRHS(-1, new VariableB_ShiftAssign { Date = prepreD, Employee = e, Group = "O" });
                        optEngine.AddRHS(-(3 - 1));
                    }
                    optEngine.CreateGreatEqual($"{ConstraintName}_a@{d:yyyy_MM_dd}@{e}");
                    ConstraintCount++;
                });
            });

            // ── C9：整月至少一次連休 2 天，否則 s^dlt = 1 ─────────────

            dataload.Employee.ForEach(e =>
            {
                dataload.Date.ForEach(d =>
                    optEngine.AddLHS(1, new VariableB_DoubleOffFlag { Date = d, Employee = e }));
                optEngine.AddLHS(2, new VariableB_DoubleOffLT2 { Employee = e });
                optEngine.AddRHS(2);
                optEngine.CreateGreatEqual($"{ConstraintName}_b@{e}");
                ConstraintCount++;
            });

            Logging.Info($"[{ConstraintName}] {ConstraintCount}");
        }
    }
}

C10 — 休假不低於平均（差距量，≥）

AVGOFF 由總班次扣掉總需求、平均到每人再保守減 1 算出，是衍生參數（在限制式裡算一次即可）。休假天數不足時，z^avg 補上差距並在目標被罰。

using OptimFoundation.Cplex;
using OptimFoundation.Core;
using HospitalRostering.Set;
using HospitalRostering.Variable;

namespace HospitalRostering.Constraint
{
    /// <summary>[C10] Σ_d y[e,d,O] + z^avg[e] ≥ AVGOFF   ∀ e</summary>

    public class Constraint_BelowAVG : ConstraintBase
    {
        private OptEngine optEngine;
        private Dataload  dataload;

        public Constraint_BelowAVG(Dataload dataload, OptEngine engine)
        { this.optEngine = engine; this.dataload = dataload; }

        {

                optEngine.AddRHS(AVGOFF);
                optEngine.CreateGreatEqual($"{ConstraintName}@{e}");
                ConstraintCount++;
            });

5.7 組裝與進入點

最後是組裝點。框架提供 OptModel——一個註冊式（Open/Closed）的 composition root：你只「登記」要用哪個 config、哪些建變數步驟、哪些建模步驟、求解成功後做什麼，OptModel 負責在對的時機把它們叫起來、計時、並自動釋放 CPLEX 資源。它對任何具體 class 零依賴，所以不需要手寫主問題類別。

框架精神：建模步驟兩用

VariableCreate 與 BuildModel 這兩個建模步驟，求解與調參兩種模式共用同一份——求解走 OptModel、掃參數走 ExperimentRunner。模型定義只寫一次，兩邊保證產生完全相同的模型。

OptModel 保證的事

① 變數一定在限制式之前建立（AddVariables 早於 AddModel）；② 內建建模 / 求解計時與整體運作時間 log；③ Execute() 回傳 bool（true = Optimal 或 Feasible），成功才跑 OnSolved；④ using 結束自動 Dispose 引擎。專案名 "HospitalRostering" 用於 log 檔命名。

第二個進入點 ExperimentRunner：同一個模型套多組 CplexConfig，每組求解一次，用框架的 Trial.Capture 記錄「完整設定快照 + 收斂數據」，累積成 Experiment 後輸出 Experiments/<name>.csv + .json。注意它重用上面同一份 VariableCreate / BuildModel。

5.8 兩種建構架構並排：手動 vs Generator＋注入 Action

到這裡你已經把整個模型建完了。但「同一份數學模型」其實可以用兩種建構架構落地——你前面跟著做的是新版（架構 B）。這一節把兩種並排，讓你看清差異、知道何時用哪個。本專案附了兩個用同一份 Model/HospitalRostering_Model.md 從頭建到尾、跑同一組 tuning 的可運作專案：

架構專案composition root變數 / 參數 class
A 手動Projects\HospitalRostering_Manual\手寫 HospitalRosteringProblem.Execute()（IDisposable）手寫屬性（: VariableBase / : ParameterBase）
B 新版（預設）Projects\HospitalRostering_Generator\Fluent OptModel（注入 Action<OptEngine>）[OptVar] / [OptParam] 由 generator 生成
關鍵：這是「兩條獨立的軸」，不是一個開關

「composition root（誰驅動引擎）」與「class 來源（誰寫樣板）」是兩條可各自選擇的軸。本教學前面其實示範的是中間最常見的組合（OptModel + 手寫 class）；架構 A、B 只是兩個端點。你完全可以漸進遷移：先把 composition 換成 OptModel，之後再把 class 換成 [OptVar]。數學模型（VariableCreate / BuildModel / 各 Constraint）兩架構逐行相同，零差異。

軸一：composition root

架構 A 自己 new / Build / Dispose 引擎，手寫主問題類別；架構 B 把這些交給框架的 OptModel，你只「注入」各建構步驟。

架構 A — HospitalRosteringProblem.cs（手動）csharp

public sealed class HospitalRosteringProblem : IDisposable
{
    private readonly Dataload _data = new();
    private OptEngine? _engine;

    public bool Execute()
    {
        var config = new CplexConfig { epGap = 0.03, timeLimit = 100, workThreads = 10 };
        _engine = new OptEngine(config);
        _engine.Build();                                // 自己管引擎生死

        new VariableCreate(_data, _engine).Build();     // ← 與 B 共用同一份

        new BuildModel(_data, _engine).Build();         // ← 與 B 共用同一份

        bool ok = _engine.Solve();
        if (ok) _data.WriteToCSV(_engine);
        return ok;
    }
    public void Dispose() => _engine?.Dispose();        // 自己負責釋放

}

7.2 調權重就是調偏好；硬約束也能軟化

本模型的「軟性偏好」其實已經內建——七個 Penalty_* 就是旋鈕。把 Penalty_SixDay 從 1.0 拉到 5.0，求解器就會更努力避免連上六天，代價是別處（如跨組別）可能多一點。不用改任何限制式，只改 Dataload 的權重。

反過來，目前的硬約束（C2 需求）若資料設成不可能滿足，模型會 infeasible。此時可把它軟化：框架的 CreateEqSoft 會自動加入正負偏差變數 並把偏差量乘 penalty 計入目標——「人力盡量補滿、真的補不到才短缺」：

Constraint_FullfillDemand.cs（soft 版）csharp

double demand  = dataload.parameter_ShiftDemand
                         .FirstOrDefault(x => x.Date == d && x.Group == g)?.QTY ?? 0;
double penalty = 100;   // 每短缺 1 人的罰金，需 >> 其它罰分才有意義


dataload.Employee.ForEach(e =>
    optEngine.AddLHS(1, new VariableB_ShiftAssign { Date = d, Employee = e, Group = g }));

// 硬：optEngine.AddRHS(demand); optEngine.CreateEqual($"...");

// 軟：允許偏離 demand，偏差量進目標式被罰

optEngine.CreateEqSoft(demand, penalty, $"{ConstraintName}@{d:yyyy_MM_dd}@{g}");
兩個前提

① 呼叫 soft 方法前，目標式必須已建立（penalty 會加進現有目標）。② penalty 要明顯大於正常罰分權重，否則求解器會「寧可付罰金也不補人」。

7.3 IIS 衝突分析：模型 infeasible 怎麼辦

排班模型最容易 infeasible 的點是需求與預排班打架：例如某天 D 班需求 5 人，卻把太多人預排成 O 或別班，使得湊不滿 5 人。Solve() 偵測到 Infeasible 時會自動跑 CPLEX 的 IIS（Irreducible Infeasible Set），找出「最小的互相矛盾限制式集合」。你只要讀回來：

    conflicts.ForEach(name => Logging.Info($"  - {name}"));

7.4 實驗環境：把調校變成可重現的數據

7.1 的旋鈕「調哪個、調多少」不該靠猜。ExperimentRunner（見 5.7）建構在框架的實驗 harness 上，把每一次求解的完整設定 + 收斂數據落地成檔，讓你用數據決策。

元件角色

new Experiment(name, desc) + .AddTrial() + .Save()累積多個 Trial，輸出 Experiments\<name>.csv + .json
Experiment.Load(name)讀回既有實驗繼續累積（同名以 RunAt + Label 去重）

每個 Trial 的 SolveMetrics 記錄：狀態、目標值、最佳 bound、MIP gap、牆鐘時間、node 數、變數 / 限制式數量；CPLEX 還會附收斂軌跡（各時間點的 incumbent 與 bound）。打開 Experiments\hospital-tuning.csv 就能一欄一欄比較哪組 config 最快把 gap 壓下來。

可重現的實驗

要讓兩次掃描可比，固定 randomSeed + parallelMode = 1（決定論），並改用 detTimeLimit（決定論 ticks）取代牆鐘 timeLimit——否則多執行緒計時本身就會抖動。掃描時 enableLog = false 並關掉 export 以加速；每個 Trial 用全新的 Dataload 與 OptEngine，避免上一輪的 Pool / 變數狀態污染下一輪。

三種調校方向的取捨

調 solver：最便宜，不動模型，先試。調權重 / 軟化限制：要改變偏好強度或讓 over-constrained 模型有解時用。改模型結構（收緊指示變數、對稱性破除、warm start）：只有在效能差很多時才動，因為改動最大、風險最高。

8常見錯誤排查

每個問題用「錯誤訊息 → 原因 → 解法」三段呈現。

① AddLHS / AddRHS 找不到變數

錯誤訊息KeyNotFoundException: AddLHS: 找不到變數 'VariableB_ShiftAssign@E1@2026-01-01@D'
原因變數類別屬性宣告順序與 BuildBVs 傳入 Set 順序不一致；名稱組成了 @E1@日期@D 而非 @日期@E1@D。也可能是該變數根本沒被 Build*Vs 建立。
解法確認屬性順序＝Build*Vs 的 Set 順序（本例 Date → Employee → Group）；確認 VariableCreate 有建這個變數。

② 限制式被靜默略過（最容易踩）

症狀限制式數量比預期少，某條規則沒生效，但沒有報錯。例如 C6 不良轉換、C5 跨組別只生效了一部分。
原因框架以「限制式名稱」去重。本模型在 foreach rule / foreach 視窗日 內若名稱沒帶上 rule 或日期索引，第 2 圈起會被當重複而跳過。
解法名稱一律帶足所有迴圈索引：C6 用 @{rule.PreGroup}_{rule.Group}、C5 用 @{cg.Group}、C4 上界用 @{sd:MM_dd}。

③ 指示變數沒被頂起來（罰分恆為 0）

症狀明明排了連六天，s^six 卻是 0、目標沒被罰。
原因只寫了上界（s ≤ ...）沒寫下界（s ≥ ...）。指示變數若只有上界，求解器會「自由地讓它 = 0」來逃避罰分。
解法「定義違規」型指示變數通常要下界（C6/C7/C8 都是 ≥）把它頂起；C4 因為要雙向鎖定才同時需要上界與下界。

④ 求解回傳 Infeasible

症狀Solve() 回傳 false，Status == Infeasible。
原因硬約束互相矛盾：常見是某日某班需求湊不滿（人都被預排走了），或方向寫反。
解法呼叫 GetConflictConstraints() 拿 IIS 衝突集合（見 7.3）；對照 .lp 檢查方向與係數；必要時把需求軟化。

⑤ 目標式沒生效 / 解恆為 0

症狀目標值是 0，或求解很快但班表明顯亂排。
原因呼叫 CreateMinimize() 前 Pool 是空的（忘了 AddLHS），或所有 Penalty_* 都設成 0。
解法確認 ObjectiveFunction.Build() 的 AddLHS 緊接在 CreateMinimize() 之前、中間沒別的 Create* 把 Pool 清掉；確認權重不全為 0。

⑥ Build 失敗：找不到 DLL / 型別

錯誤訊息error CS0246: 找不到型別或命名空間名稱 'OptEngine' / 參考警告 HintPath 無效
原因csproj 的 HintPath 相對路徑層數算錯，或 dlls\ 缺檔。
解法Projects\專案\ 下用 ..\..\dlls\；確認五個 DLL 都在；namespace 用 OptimFoundation.Cplex 與 OptimFoundation.Core。

⑦ 求解慢到天荒地老

症狀跑很久、Gap 卡在某個百分比下不來。
原因排班是大型 0/1 問題，要求證明到 0% 最佳很貴。
解法放寬 epGap（如 0.03）、設 timeLimit、mipEmphasis = 1 先求可行解；班表「夠好」通常就能用。

10與 AI 協作開發

OptimFoundation 從設計上就是為了「人給題目、AI 寫模型」。框架的每一條慣例（天條、兩階段、Pool、資料夾規範）都是為了讓 AI 產出的 code 穩定、可審閱、可重現。這一章講怎麼把 AI 當成你的建模工程師。

10.1 為什麼這個框架適合交給 AI

框架特性對 AI 協作的意義
兩階段（先數學模型、後 code）AI 先輸出可審閱的數學模型，人確認後才轉 code，錯誤早抓
QTY 天條（數值全進 Parameter）AI 不會把魔術數字散落在邏輯裡；改數據不必改 code
Pool 機制 + 不可移項規則AI 逐條把數學式機械式翻成 AddLHS/AddRHS，不需「理解」也不會寫錯方向
資料夾規範（claudemdTemplate）每個資料夾一份 CLAUDE.md＝AI 的 few-shot 規範，產出一致
source generator（[OptVar]/[OptParam]）AI 用一行 attribute 宣告變數/參數，樣板由編譯器補，產出更短更不易錯
OptModel 雙模式 + ExperimentRunnerAI 產的專案天生可求解 + 可掃參數，tuning 也能交給 AI

10.2 標準協作流程

1
給題目：用自然語言描述決策、目標、限制與數值來源。越具體越好——尤其「每個數字從哪來」。
2
要 AI 先出數學模型（Model/*.md）：Sets / Parameters / Variables / Objective / Constraints，限制式編號 [C1][C2]…。先確認數學，再讓它寫 code。
3
AI 依資料夾規範產 code：Parameter → Set(Dataload) → Variable → Objective → Constraint → Program。每條限制式對應模型的一個 [Cx]。
4
確認 Dataload 數值：第二個暫停點。數值必須與題目完全一致，AI 不得自行推算或填佔位符。
5
build & fix：AI 跑 dotnet build，依 compiler error 修對應 .cs，循環到綠。
6
tuning：dotnet run -- experiment 掃參數，AI 讀 Experiments/*.csv 給調校建議。
兩個暫停確認點（最重要）

數學模型（步驟 2）與 Dataload 數值（步驟 4）是最容易出錯的兩處。讓 AI 在這兩點停下來給你看，確認後再往下，能擋掉九成「跑得出來但答案不對」。
