讓電腦幫你做「最好的決定」

你不用懂數學、也不用會寫程式。這份教學會用一個小工廠的故事，帶你一步步理解 OptimFoundation 在做什麼，最後親手寫出一個真的能跑、會算出「最賺錢方案」的小程式。

不需要數學背景 不需要程式背景 看故事學概念 照著抄就能跑

讀完這份你會有什麼

  • 知道「最佳化（optimization）」到底在解什麼問題、什麼時候該用它。
  • 學會把任何「在限制下做最好選擇」的問題，拆成 4 個零件。
  • 看懂 OptimFoundation 的每個檔案在做什麼——它其實只是把那 4 個零件「翻譯」成程式。
  • 親手做出一個會算出最佳生產計畫的小專案。
怎麼讀

從上到下讀就好，每一節都接著前一節。看到程式碼先別緊張——我會先用白話講「這在幹嘛」，再一行一行解釋。你只要會複製貼上，就能跟著做完。

1什麼是「最佳化」？先講一個你一定遇過的情況

想像你開了一間小玻璃工廠，可以做兩種玻璃：普通玻璃和強化玻璃。

  • 強化玻璃比較好賺（一片賺 10 元），普通玻璃賺得少一點（一片賺 8 元）。
  • 但每片玻璃都要先用加熱機烤、再用冷卻機冷卻，而這兩台機器每天能用的時間是有限的。
  • 強化玻璃比較「吃機器時間」，做一片要佔用機器比較久。

問題來了：今天這兩種玻璃各做幾片，才能賺最多錢？

「那就全做強化玻璃啊，它比較賺！」——可是強化玻璃太吃機器時間，全做強化的話機器很快就滿了，總片數做不多。「那全做普通？」——普通片數能做多，但每片賺得少。最好的答案通常是某種「混搭」，而且當產品變多、限制變複雜時，用猜的幾乎不可能猜到最好。

這就是「最佳化」

當你要在一堆限制底下，做出一個讓某個目標（賺最多、花最少、跑最快）達到最好的決定——這就是一個最佳化問題。OptimFoundation 就是幫你把這種問題交給電腦，由電腦在所有可能方案裡找出「保證最好」的那一個。

聽起來很抽象嗎？別擔心。下一節我們會發現：不管問題長怎樣，它永遠由同樣的 4 個零件組成。

2任何最佳化問題，都只有 4 個零件

這是整份教學最重要的一張圖。把它記住，你以後看任何最佳化問題都能秒拆解：

① 你要「決定」什麼？
那些你還不知道、要由你（或電腦）拍板的數字。玻璃工廠裡就是：普通玻璃做幾片、強化玻璃做幾片。
決策變數 Variables
② 有哪些「固定」的數字？
事先就知道、不會變的已知條件。例如：每片玻璃賺多少、要佔多少機器時間、機器總共能用多久。
參數 Parameters
③ 你希望什麼「最好」？
你想最大化或最小化的那一件事。這裡是：總利潤最大。
目標式 Objective
④ 有哪些「不能違反」的規則？
再怎麼想賺，也不能超過的底線。例如：用掉的加熱機時間不能超過上限、冷卻機也是。
限制式 Constraints

用一句話串起來：在「限制」之內，調整你能「決定」的東西，讓「目標」最好。 那些「固定的數字」則是計算時的依據。

把玻璃工廠套進這 4 個零件

① 決定：普通做幾片、強化做幾片。
② 固定數字：普通賺 8、強化賺 10；普通要加熱 3 分鐘・冷卻 5 分鐘，強化要加熱 5 分鐘・冷卻 8 分鐘；加熱機共 300 分鐘、冷卻機共 300 分鐘。
③ 目標：總利潤最大。
④ 限制：所有玻璃用掉的加熱時間 ≤ 300；冷卻時間 ≤ 300。

所以才需要求解器（solver）

求解器就是一個數學引擎，它能在所有合法方案裡，保證找出目標最好的那一組，而且很快。OptimFoundation 內建的引擎叫 CPLEX。你的工作不是去解，而是把這 4 個零件「描述」清楚，剩下交給它。

5跟著做：把玻璃工廠寫成程式

下面每一步都長這樣：先白話講這步在幹嘛 → 給程式碼 → 解釋每行。你只要新增對應的檔案、把程式碼貼進去即可。專案放在 Projects/GlassFactory/。

開始前（環境）

你需要安裝 .NET 8，並且專案能參考到框架的 DLL（已經放在 ClaudeAIAssistant/dlls/）。環境細節在主教學裡有，這裡先假設已就緒，把重點放在「怎麼把問題寫出來」。每個資料夾的開頭都有一行 namespace，照抄即可。

1

第 ② 個零件：把「固定的數字」變成一個類別

每一種玻璃都有 3 個固定數字：利潤、加熱時間、冷卻時間。我們先做一個「資料表的一列長什麼樣」的模板，叫 Parameter_GlassSpec。它就是一張表的欄位定義。

Parameter/Parameter_GlassSpec.cscsharp
using OptimFoundation.Core;

namespace GlassFactory.Parameter
{
    // 一種玻璃的規格＝表格的一列。每個玻璃會有一筆這樣的資料。
    public class Parameter_GlassSpec : ParameterBase
    {
        public string GlassType   { get; set; } = string.Empty;  // 玻璃名稱
        public double HeatingTime { get; set; }                  // 每片要加熱幾分鐘
        public double CoolingTime { get; set; }                  // 每片要冷卻幾分鐘
        public double Profit      { get; set; }                  // 每片賺多少
    }
}

逐行看：

  • : ParameterBase＝「我是一筆參數資料」，繼承框架的基底類別，照抄即可。
  • 中間 4 行就是表格的 4 個欄位。{ get; set; } 是 C# 寫「這是一個可讀可寫的欄位」的固定寫法，照抄。
  • 這一步只是「定義欄位」，還沒填數字。填數字在下一步。
2

填進真正的數字，並順便生出「有哪些玻璃」

Dataload 是整個專案唯一放實際數字的地方。我們在這裡填兩種玻璃的規格、兩台機器的產能上限。

Set/GlassDataload.cscsharp
using OptimFoundation.Cplex;
using OptimFoundation.Core;
using GlassFactory.Parameter;
using GlassFactory.Variable;

namespace GlassFactory.Set
{
    public class GlassDataload
    {
        // 兩種玻璃的固定數字（對照第 3 節那張表）
        public List<Parameter_GlassSpec> parameter_GlassSpec = new()
        {
            new() { GlassType = "Regular",  HeatingTime = 3, CoolingTime = 5, Profit = 8  },
            new() { GlassType = "Tempered", HeatingTime = 5, CoolingTime = 8, Profit = 10 }
        };

        // 兩台機器的總時間上限
        public double HeatingCapacity = 300;
        public double CoolingCapacity = 300;

        // 「有哪些玻璃」＝把上面每一列的名字撈出來，不用另外手打
        public List<string> GlassTypes => parameter_GlassSpec.Select(s => s.GlassType).ToList();
    }
}

逐行看：

  • List<Parameter_GlassSpec>＝「一串 GlassSpec」，也就是一張表，裡面塞了兩列（Regular、Tempered）。數字就照第 3 節那張表填。
  • Regular／Tempered 就是「普通」「強化」的英文，當作名稱用。
  • 兩個 Capacity = 300 是機器上限。
  • 最後 GlassTypes 這一行：自動把「Regular、Tempered」這份清單湊出來——之後建立決策變數要用。你不用手動再列一次名字。
框架天條：數字只能放這裡

所有數字（利潤、時間、上限）一律放在 Parameter / Dataload。絕對不要把數字直接寫死在後面的目標式或限制式裡。這樣以後要改數字，只改這一個檔，不會出錯。

3

第 ① 個零件：宣告「要決定的東西」

我們要決定「每種玻璃做幾片」。所以建立一個決策變數 VariableX_Production，它「依玻璃種類」而存在——Regular 一個、Tempered 一個。

Variable/VariableX_Production.cscsharp
using OptimFoundation.Core;

namespace GlassFactory.Variable
{
    // 決策變數：某種玻璃要生產的片數（由求解器決定）
    public class VariableX_Production : VariableBase
    {
        public string GlassType { get; set; } = string.Empty;  // 這個變數對應哪種玻璃
    }
}

重點：

  • 名稱開頭的 VariableX_ 的 X 代表「連續變數」——可以是 12.5 這種小數。（玻璃片數其實該是整數，但入門先用連續，第 7 節會說明 X / I / B 的差別。）
  • 裡面只放「索引」（這變數是哪種玻璃），不放數字。片數是多少，是求解器算出來的，不是你填的。

然後告訴框架「請依玻璃清單，把這些變數建出來」：

Variable/VariableCreate.cscsharp
using OptimFoundation.Cplex;
using OptimFoundation.Core;
using GlassFactory.Set;

namespace GlassFactory.Variable
{
    public class VariableCreate
    {
        private OptEngine     optEngine;
        private GlassDataload dataload;

        public VariableCreate(GlassDataload dataload, OptEngine engine)
        {
            this.optEngine = engine;
            this.dataload  = dataload;
        }

        public void Build()
        {
            // 依玻璃清單建立「連續變數」：BuildCVs 的 C = Continuous
            optEngine.BuildCVs<VariableX_Production>(dataload.GlassTypes);
        }
    }
}

這段大部分是固定樣板（建構子那段照抄）。真正關鍵只有一行：BuildCVs<VariableX_Production>(dataload.GlassTypes)——意思是「照著玻璃清單，每種玻璃各生一個連續變數」。於是求解器手上就有了 Regular 片數 和 Tempered 片數 兩個待決定的數字。

4

第 ③ 個零件：寫出「目標」——總利潤最大

總利潤 ＝ 8 × 普通片數 ＋ 10 × 強化片數。我們把它「一片一片地疊加」起來，最後說一句「請最大化」。

Objective/ObjectiveFunction.cscsharp
using OptimFoundation.Cplex;
using OptimFoundation.Core;
using GlassFactory.Set;
using GlassFactory.Variable;

namespace GlassFactory.Objective
{
    public class ObjectiveFunction
    {
        private OptEngine     optEngine;
        private GlassDataload dataload;

        public ObjectiveFunction(GlassDataload dataload, OptEngine engine)
        {
            this.optEngine = engine;
            this.dataload  = dataload;
        }

        public void Build()
        {
            // 對每一種玻璃：把「利潤 × 該玻璃片數」加進目標
            dataload.parameter_GlassSpec.ForEach(spec =>
                optEngine.AddLHS(spec.Profit, new VariableX_Production { GlassType = spec.GlassType }));

            optEngine.CreateMaximize();   // 收尾：請求解器「最大化」這個總和
        }
    }
}

白話： ForEach 是「對清單裡每一筆做同一件事」。對每種玻璃，AddLHS(利潤, 該玻璃的片數變數) 就是把「利潤 × 片數」這一項丟進一個暫存籃子。兩種玻璃都丟完，籃子裡就是 8×Regular + 10×Tempered。最後 CreateMaximize() ＝「結帳：請最大化籃子裡的東西」。

這個「丟籃子 → 結帳」就是 Pool 機制第 7 節會再講，現在先記住這個畫面就好。

5

第 ④ 個零件：寫出「限制」——機器時間不能超過

限制一：所有玻璃用掉的加熱時間 ≤ 300。和目標一樣用「丟籃子」，只是這次結帳說的是「小於等於 300」。

Constraint/Constraint_Heating.cscsharp
using OptimFoundation.Cplex;
using OptimFoundation.Core;
using GlassFactory.Set;
using GlassFactory.Variable;

namespace GlassFactory.Constraint
{
    // [限制一] 加熱機產能：Σ (加熱時間 × 片數) ≤ 300
    public class Constraint_Heating : ConstraintBase
    {
        private OptEngine     optEngine;
        private GlassDataload dataload;

        public Constraint_Heating(GlassDataload dataload, OptEngine engine)
        {
            this.optEngine = engine;
            this.dataload  = dataload;
        }

        public void Build()
        {
            // 左邊：每種玻璃的「加熱時間 × 片數」全部加起來
            dataload.parameter_GlassSpec.ForEach(spec =>
                optEngine.AddLHS(spec.HeatingTime, new VariableX_Production { GlassType = spec.GlassType }));

            optEngine.AddRHS(dataload.HeatingCapacity);  // 右邊：上限 300
            optEngine.CreateLessEqual(ConstraintName);   // 收尾：左邊 ≤ 右邊
        }
    }
}

白話： 左邊（AddLHS）疊加 3×Regular + 5×Tempered；右邊（AddRHS）放上限 300；CreateLessEqual ＝「結帳：左邊要 ≤ 右邊」。ConstraintName 是這條限制的名字，照抄即可。

限制二（冷卻）幾乎一模一樣，只是把 HeatingTime 換成 CoolingTime、產能換成冷卻機：

Constraint/Constraint_Cooling.cs（與上面只差兩個字）csharp
        public void Build()
        {
            dataload.parameter_GlassSpec.ForEach(spec =>
                optEngine.AddLHS(spec.CoolingTime, new VariableX_Production { GlassType = spec.GlassType }));

            optEngine.AddRHS(dataload.CoolingCapacity);
            optEngine.CreateLessEqual(ConstraintName);
        }
方向不能亂猜

數學寫 ≤ 就用 CreateLessEqual、≥ 用 CreateGreatEqual、= 用 CreateEqual。照你在第 2、3 節想清楚的限制方向寫，不要反過來。

6

把零件組裝起來

用一個 BuildModel 把「目標 + 兩條限制」依序叫起來（這只是個召集人）：

Constraint/BuildModel.cscsharp
public void Build()
{
    new ObjectiveFunction(dataload, engine).Build();   // ③ 目標
    new Constraint_Heating(dataload, engine).Build();  // ④ 限制一
    new Constraint_Cooling(dataload, engine).Build();  // ④ 限制二
}
7

進入點：按下「開始算」

最後 Program.cs 把一切串起來、按下執行。OptModel 像一條生產線，你只是登記各步驟，最後 Execute() 會自動「建變數 → 建模型 → 求解 → 輸出結果」。

Program.cscsharp
using OptimFoundation.Cplex;
using GlassFactory.Set;
using GlassFactory.Variable;
using GlassFactory.Constraint;

var dataload = new GlassDataload();   // 載入所有固定數字

using (var m = new OptModel("GlassFactory")
    .UseConfig(() => new CplexConfig { timeLimit = 60, enableLog = true })
    .AddVariables(e => new VariableCreate(dataload, e).Build())  // ① 先建變數
    .AddModel(e => new BuildModel(dataload, e).Build())          // ③④ 再建目標+限制
    .OnSolved(e => dataload.WriteToCSV(e)))                      // 算完後印結果
{
    bool ok = m.Execute();   // ← 真正開始計算
    if (!ok) Console.Error.WriteLine("[ERROR] 求解失敗");
}

注意順序：一定先建變數（AddVariables），再建目標和限制（AddModel）——因為目標和限制會用到變數。框架已經幫你固定好這個順序，照抄就對。

7三個你一定要懂的小觀念

觀念一：Pool 機制——「丟籃子 → 結帳」

你已經用過了。建一條式子時，先用 AddLHS（左邊）和 AddRHS（右邊）把各項逐一丟進暫存籃子，最後呼叫一個 Create… 來結帳：

結帳用的方法意思
CreateMaximize() / CreateMinimize()這是目標：最大化 / 最小化籃子裡的總和
CreateLessEqual(名字)這是限制：左邊 ≤ 右邊
CreateGreatEqual(名字)這是限制：左邊 ≥ 右邊
CreateEqual(名字)這是限制：左邊 ＝ 右邊

結帳完，籃子自動清空，下一條式子重新開始。所以你可以在迴圈裡一直 AddLHS 疊加很多項，這在「很多產品、很多限制」時超好用。

觀念二：三種「要決定的東西」——X / I / B

回想「玻璃片數其實不該是小數」。決策變數依「能取什麼值」分三種，用類別名稱開頭區分：

開頭類型能取的值建立方法適合
VariableX_連續 Continuous任意 ≥ 0 的數（可小數）BuildCVs重量、時間、金額
VariableI_整數 Integer0,1,2,3…BuildIVs「幾片」「幾個人」
VariableB_二元 Binary只有 0 或 1BuildBVs「要/不要」「開/關」的是非題

我們的玻璃片數，嚴謹一點應該用 VariableI_（整數）。入門用連續是為了簡單；之後遇到「某件事要不要做」這種是非題，就會用到 B（二元）——那是進階教學裡醫院排班的主角。

觀念三：先想清楚數學，再寫程式

這份教學的順序不是偶然：先把 4 個零件用白話／表格想清楚（第 2、3 節），才動手寫程式（第 5 節）。框架刻意這樣設計——程式只是把想清楚的東西「翻譯」出來。如果你跳過思考直接寫，很容易寫出「跑得出來但答案是錯的」程式。

記住這條鐵則

在 4 個零件還沒想清楚前，不要急著寫 .cs。想清楚了，寫程式就只是照抄。

8卡住了？最常見的幾種狀況

編譯失敗：找不到 DLL / 型別

錯誤訊息：類似 error CS0246: 找不到型別或命名空間名稱 'OptEngine'。
原因：專案沒參考到框架 DLL，或相對路徑層數不對。
解法：檢查 .csproj 裡 DLL 的路徑 ..\..\dlls\ 是否對應你專案的實際層數，並確認檔名拼對。

執行說「求解失敗 / Infeasible」

原因：你的限制互相矛盾，根本沒有任何方案能同時滿足（例如要求做超過機器能負荷的量）。
解法：回到第 2、3 節，逐條檢查限制方向（≤ 還是 ≥）有沒有寫反、上限數字是否填錯。這幾乎都是「人想錯」而不是程式 bug。

答案明顯不對（例如全是 0）

原因：多半是目標寫反（該 Maximize 卻 Minimize），或某個係數從 Parameter 取錯。
解法：檢查 ObjectiveFunction 用的是不是 CreateMaximize()；檢查 AddLHS 裡放的係數（利潤、時間）是否對應正確的欄位。

變數忘了建就用

原因：在 Program.cs 把 AddModel 寫在 AddVariables 前面。
解法：順序一定是先 AddVariables、後 AddModel——目標和限制都會用到變數，變數必須先存在。

9恭喜！你已經會基本套路了

你剛剛完整走過一個最佳化問題：看懂問題 → 拆成 4 個零件 → 翻譯成檔案 → 跑出保證最好的答案。這個套路對所有問題都一樣，差別只在零件的多寡與複雜度。
