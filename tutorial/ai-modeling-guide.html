AI-Assisted Optimization Modeling · 開發整合手冊

用 AI 把問題變成可跑、可調的最佳化模型

這份手冊整合 OptimFoundation 的全部開發面資訊，串成一條人機協作流水線： 人給題目 → AI 產數學模型 → AI 機械式翻成 C# → 求解 → 調校。 框架的每一條慣例都是為了讓 AI 的產出穩定、可審閱、可重現。

給會基本程式的人 Modeling → Coding → Tuning .NET 8 · CPLEX Fluent OptModel + Source Generator ExperimentRunner
這份手冊的定位

這是開發者導向的整合總覽。完全零基礎請先看 新手村（一個玻璃工廠的小 LP）；想看一個大型實戰，看 醫院排班 MILP。本頁把「怎麼用 AI 走完整個開發流程」一次講清楚，並標出每一塊的單一來源規範。

0核心理念：為什麼這個框架天生適合交給 AI

OptimFoundation 不是「另一個求解器包裝」。它的設計目標是：把最佳化開發拆成 AI 能穩定、機械式產出的步驟。每條慣例都對應一個讓 AI 不易出錯的理由：

框架特性對 AI 協作的意義
兩階段（先數學模型、後 code）AI 先輸出可審閱的數學模型，人確認後才轉 code，錯誤在最便宜的階段就被抓出來
QTY 天條（數值全進 Parameter）AI 不會把魔術數字散落在邏輯裡；改數據不必改 code，也讓人一眼能核對數值
Pool 機制 + 不可移項規則AI 把數學式逐項機械式翻成 AddLHS/AddRHS，不需「理解」也不會寫錯方向
資料夾規範（claudemdTemplate）每個資料夾一份 CLAUDE.md ＝ AI 的 few-shot 範本，產出一致
Source generator（[OptVar]/[OptParam]）AI 用一行 attribute 宣告變數/參數，樣板由編譯器補，產出更短、更不易錯
OptModel 雙模式 + ExperimentRunnerAI 產的專案天生可求解 + 可掃參數，連 tuning 都能交給 AI
一句話記住

你的工作是把問題說清楚、在兩個關卡確認；AI 的工作是照框架慣例機械式產出。框架負責讓這條分工穩定成立。

Phase 11Modeling：自然語言 → 數學模型

目標：把自然語言的問題，變成無歧義、可被機器解析的數學模型。這一步只動筆、不動 code。產物是 Model/ProjectName_Model.md，裡面五區塊：Sets → Parameters → Variables → Objective → Constraints，限制式編號 [C1][C2]… 之後與程式一一對應。

天條（從 Modeling 就開始）

模型裡所有數值（係數、上下限、比例、罰分權重）都必須對應到某個 Parameter.QTY，禁止裸數字。這條會一路貫穿到 code。

1.2 Stage 1–3：逐步結構化 + 約束分類

  • SimpleModel：用結構化自然語言列出「決策 / 目標 / 限制 / 已知數據」。
  • StandardModel：統一術語與量綱，並把每條約束分類（見下表）。
  • KeyInfo：抽成 JSON 骨架（Sets、Parameters、Variables、Objective、Constraints），供後續機器解析。

約束分類會直接影響後面 code 的建模手法（例如 Implication 要用指示變數 + Big-M）：

類型語言線索
UB / LB 上下界"at most" / "at least" / "no more than"
Balance 平衡"must equal" / "total in = total out"
Proportional 比例"in proportion to" / "at least X times"
Conjunction 合取"only if all"
Disjunction 析取"at least one of"
Exclusive XOR"exactly one of"
Implication 蘊含"if A then B"
Conditional Activation"only if" / Big-M 條件啟動

1.3 Stage 4 / 4b：AML 數學模型與驗證

用 AMPL 命名慣例 + Markdown 撰寫，數學式一律用 LaTeX，五區塊齊全。命名規則：

  • 所有 identifier 用 PascalCase（ProductIndex、TotalCost），不隨意複數化；Constraint 名稱即未來的 Constraint_<Name> 類別名。
  • 線性化是硬規則——禁止 x*y、x/y、abs/min/max/if；比例與邏輯條件要先線性化（Big-M、比例上下界）。

Stage 4b 驗證：對照原始描述檢查線性性、命名一致、Sets/Params/Vars 是否齊全 —— 這是進入 Coding 前的最後數學關卡。

暫停確認點 ① 數學結構

AML 數學模型是最容易出錯、也最便宜修正的地方。讓 AI 在這裡停下來給你看，確認 Sets / 變數型態 / 目標方向 / 每條限制都對，再往下。這一關擋掉大半「跑得出來但答案不對」。

1.4 範例：把玻璃工廠寫成數學模型

玻璃工廠在加熱機與冷卻機的產能限制下，決定各類玻璃生產量以最大化利潤（這是一個 LP）。

Sets / Parameters / Variables：

I 玻璃種類集合 h[i], c[i] 玻璃 i 的加熱 / 冷卻時間（分鐘/片） p[i] 玻璃 i 的單位利潤 H, C 加熱機 / 冷卻機 總產能上限 x[i] ≥ 0 生產玻璃 i 的片數（連續變數）, ∀ i ∈ I

Objective 與 Constraints：

max Σ_{i∈I} p[i] · x[i] 目標：總利潤最大 [C1] Σ_{i∈I} h[i] · x[i] ≤ H 加熱機產能 [C2] Σ_{i∈I} c[i] · x[i] ≤ C 冷卻機產能

到這裡模型就定稿了。Phase 2 只是把這幾條機械式翻成 code，沒有任何自由發揮。如果 coding 時發現歧義，要回來補這份模型，而不是在 code 裡臨時假設。

Phase 22Coding：模型機械式翻成可跑的 C#

核心心法：code 只是翻譯，不是創作。AML 的每一條 [Cx] 對應一個 Constraint_Xxx，每個零件對應一種檔案。

2.2 兩種建模架構：source generator（預設）vs 手寫（後路）

預設：用 [OptVar]/[OptParam] attribute 宣告，class body（properties、QTY）由 AutoSetsGenerator 在編譯期生成——最短、最不易錯：

預設：source generator（partial class + attribute）csharp

[OptParam("Set1", "Set2:double", "Set3:int", "Set4:DateTime")]
public partial class Parameter_Template { }

// 變數：宣告型別與 set
[OptVar(VarType.Binary, "A")]
public partial class VariableB_A { }

後路：需要逐行掌控時，手寫 properties（兩種等價，產出的變數完全一樣）：

後路：手寫 propertiescsharp
public class Parameter_ShiftDemand : ParameterBase

    public string   Group { get; set; } = string.Empty;

}

兩架構並排對照見 醫院排班教學 §5.8；範例專案 Projects/HospitalRostering_Generator（預設）vs HospitalRostering_Manual（後路）。

2.3 Pool 機制：AddLHS / AddRHS / Create*（整個框架最核心）

你不直接寫出 3x + 2y ≤ 10，而是把一條式子逐項累加進一個暫存區（Pool），最後呼叫 Create* 收尾並清空：

AddLHS(coef, varObj) ← 左側變數項 AddLHS(constant) ← 左側常數項（若有） AddRHS(value) ← 右側常數 AddRHS(coef, varObj) ← 右側變數項（若有） CreateLessEqual(name) ← ≤ 建立並清空 Pool CreateGreatEqual(name) ← ≥ CreateEqual(name) ← = CreateMaximize() ← 目標：最大化 Pool（或 CreateMinimize） 區間用 CreateRange(lb, ub, name)；軟性用 CreateLeSoft / GeSoft / EqSoft
不變式（最重要的硬規則，AI 與人都不能破）

① AML 左側的項 → AddLHS；右側的項 → AddRHS。
② 嚴禁移項、改號、合併化簡、翻轉比較方向。數學寫 ≤ 就 CreateLessEqual。
③ 參數值先用 LINQ 查進區域變數，再傳入 AddLHS/AddRHS（不可把 LINQ 內嵌進參數）。

Pool 典型寫法（一條限制式）csharp
public void Build()
{
    dataload.Set1.ForEach(i =>
    {
        // 先把參數值 LINQ 查進區域變數
        var coef = dataload.parameter_Coef
            .FirstOrDefault(x => x.Set1 == i)?.QTY ?? 0.0;

        engine.AddLHS(coef, new VariableX_Amount { Set1 = i });   // 左側逐項累加
        engine.AddRHS(rhsValue);                                  // 右側常數
        engine.CreateLessEqual($"{ConstraintName}@{i}");          // 收尾 ≤，Pool 自動清空
        ConstraintCount++;
    });
    Logging.Info($"[{ConstraintName}] {ConstraintCount}");
}

2.4 Dataload、組裝、與雙模式進入點

Dataload 是唯一存放實際數字的地方，數值保真：與原始問題描述完全一致，不四捨五入、不推算、不填佔位符。Sets 由 Parameters 衍生，scalar 參數也用 List<Parameter_X> 只放一筆。

VariableCreate（依型別建所有變數）與 BuildModel（先 Objective、再各 Constraint）是兩個 step class，被 solve 與 experiment 兩模式共用。Program.cs 用 Fluent OptModel 串起來，並支援雙模式：

Program.cs（唯一進入點 · solve / experiment 雙模式）csharp
using System.Linq;
using OptimFoundation.Cplex;
using ProjectName.Set;
using ProjectName.Variable;
using ProjectName.Constraint;

if (args.Contains("experiment")) { ExperimentRunner.Run(); return; }   // tuning 模式

var dataload = new ProjectNameDataload();
using (var m = new OptModel("ProjectName")
    .UseConfig(() => new CplexConfig { epGap = 0.0, timeLimit = 60, workThreads = 4, enableLog = true, exportSol = true })
    .AddVariables(e => new VariableCreate(dataload, e).Build())   // ① 先建變數
    .AddModel(e => new BuildModel(dataload, e).Build())           // ②③ 再建目標+限制
    .OnSolved(e => dataload.WriteToCSV(e)))                       // 求解成功後輸出
{
    bool ok = m.Execute();   // 建模 → 求解 → OnSolved → 自動 Dispose
}

OptModel 保證 AddVariables 先於 AddModel（目標/限制會用到變數）、內建 build/solve 計時、並自行 Dispose engine。

2.5 完整可跑範例：玻璃工廠（對照 1.4 的數學模型）

每個零件一個檔，全部直接對應到 Phase 1 的模型。先是參數與資料：

Parameter/Parameter_GlassSpec.cs · Set/GlassDataload.cscsharp
public class Parameter_GlassSpec : ParameterBase   // 一種玻璃 = 一列
{
    public string GlassType   { get; set; } = string.Empty;
    public double HeatingTime { get; set; }
    public double CoolingTime { get; set; }
    public double Profit      { get; set; }
}

public class GlassDataload
{
    public List<Parameter_GlassSpec> parameter_GlassSpec = new()   // 數值保真
    {
        new() { GlassType = "Regular",  HeatingTime = 3, CoolingTime = 5, Profit = 8  },
        new() { GlassType = "Tempered", HeatingTime = 5, CoolingTime = 8, Profit = 10 }
    };
    public double HeatingCapacity = 300;   // H
    public double CoolingCapacity = 300;   // C
    public List<string> GlassTypes => parameter_GlassSpec.Select(s => s.GlassType).ToList();  // Set I 由參數衍生
}

變數（連續，對應 x[i] ≥ 0）：

Variable/VariableX_Production.cs · Variable/VariableCreate.cscsharp
public class VariableX_Production : VariableBase
{
    public string GlassType { get; set; } = string.Empty;   // 只放索引，不放數值
}

// VariableCreate.Build()：依玻璃清單建立連續變數
optEngine.BuildCVs<VariableX_Production>(dataload.GlassTypes);

目標（max Σ p[i]·x[i]）與限制（[C1] 加熱）——注意都是 Pool「逐項累加 → 收尾」：

Objective/ObjectiveFunction.cs · Constraint/Constraint_Heating.cscsharp
// 目標：max Σ Profit[i]·x[i]
dataload.parameter_GlassSpec.ForEach(spec =>
    optEngine.AddLHS(spec.Profit, new VariableX_Production { GlassType = spec.GlassType }));
optEngine.CreateMaximize();

// [C1] 加熱機產能：Σ Heating[i]·x[i] ≤ H
dataload.parameter_GlassSpec.ForEach(spec =>
    optEngine.AddLHS(spec.HeatingTime, new VariableX_Production { GlassType = spec.GlassType }));
optEngine.AddRHS(dataload.HeatingCapacity);
optEngine.CreateLessEqual(ConstraintName);
// [C2] 冷卻只差：HeatingTime→CoolingTime、HeatingCapacity→CoolingCapacity

組裝（BuildModel 先目標、再限制）後，dotnet run 會算出：Regular 60 片、Tempered 0 片、最大利潤 $480——比較貴的強化玻璃因太吃機器時間，最佳解反而一片都不做。

2.6 Stage 14：build & fix 循環 + 暫停確認點 ②

dotnet build → 失敗 → 擷取 compiler error → 修對應 .cs → 重試（最多 5 次） → 5 次全失敗 → 標記 Failed，保留 error log
暫停確認點 ② Dataload 數值

數學結構（點①）與 Dataload 數值是兩個最易錯處。讓 AI 在填完 Dataload 後停下來，逐筆核對數值與題目一致，AI 不得自行推算或填佔位符。確認後再 build。

3正確性 Gate：沒過不准進 Tuning

求解成功 ≠ 答案正確。進 Tuning 前，先驗：

  • 目標值對得上問題描述的量級嗎？（手算一兩個小案例對照）
  • 解滿足所有限制、且語意合理嗎？（沒有「負產量」「該滿足的需求沒滿足」）
  • 若 Infeasible：模型不可行，回 Coding/Modeling，用 IIS 衝突分析定位（見 §6），不要急著調參數。

沒過就回 Phase 1/2 修正。對錯誤的模型做加速，只會更快得到錯的答案。

Phase 34Tuning：讓它跑得快又收斂

黃金順序

先做模型優化，再調 solver 旋鈕。 模型改一刀的效益，通常遠大於調十個旋鈕。tuning 只改求解策略、不改數學意義（同樣禁止移項/改號/翻轉/四捨五入）。

先判斷觸發來自哪一層：

觸發線索入口
structure（數學結構）加/刪約束、改 Big-M、加 valid inequality回 Modeling（Stage 4）重跑
data（數值）改參數值改 Dataload
solver（求解參數）timeout / gap 過大 / 太慢但模型正確調 CplexConfig 旋鈕

4.1 模型層優化（最高優先）

  • 變數型態：求解難度 CV ≪ IV < BV。盡量用自由度最高者；能線性化或取整就用連續。網路流 / 指派且具 totally unimodular 結構 → LP 鬆弛即整數解，直接用 CV。
  • 模型精簡：聚合同型約束、刪冗餘（dominated）約束、收緊變數上下界（BuildCVs<>(lb, ub, ...)）。
  • Big-M 取最小可行值：M 過大會使 LP 鬆弛鬆散、節點爆增。
  • 對稱性消除：對等價變數加排序約束（x₁ ≥ x₂ ≥ …），打斷等價分支。
  • Warm start：高品質初始解大幅減少 B&B 節點（Foundation 目前無 MIP start API，等效用 mipEmphasis=1 / rinsHeur / HeuristicEffort 讓 CPLEX 自己快點找 incumbent）。

4.2 Solver 層調校（CplexConfig 旋鈕）

CplexConfig 欄位 null = 用 CPLEX 預設，tuning 只設要動的。常見症狀對應動作：

症狀動作
timeout 但正確提 timeLimit / mipEmphasis=1 / 開 parallelMode；連續放寬仍 timeout → 回 structure reformulation
gap 過大加切割（gomoryCuts/mirCuts）→ nodeSelect=1(best-bound) → 收 epGap / mipEmphasis=2
急著找可行解nodeSelect=0(DFS) → intSolLimit/nodeLimit → 放寬 epGap → mipEmphasis=1
記憶體爆treeMemoryLimit + nodeFileInd=2/3
數值不穩numericalEmphasis=true
要可重現實驗parallelMode=1 + 固定 randomSeed + detTimeLimit
記憶體旋鈕的順序 gotcha

Configuration() 在設 workMemory 時會強制 MIP.Strategy.File=0。若要「記憶體爆 → 溢寫節點檔」，必須在設 workMemory 之後再設 nodeFileInd=2/3，否則被覆蓋。

4.3 用 ExperimentRunner 實際掃描（可執行架構）

每個專案標配 ExperimentRunner.cs，建構於框架 harness（Experiment / Trial.Capture / ITunableConfig）：

PowerShellpowershell

  • 抽象旋鈕（跨引擎）：Emphasis / Seed / FeasibilityTol / OptimalityTol / RootAlgorithm / Presolve / MemoryLimitMb。
  • CPLEX 專屬：直接設 camelCase 欄位 varSel / nodeSelect / workThreads / gomoryCuts / mirCuts …。

7API / 天條速查

建模窗口（engine API）csharp
using OptimFoundation.Core;
using OptimFoundation.Cplex;

// 建變數
engine.BuildCVs<VariableX_Name>(setA, setB);      // 連續
engine.BuildIVs<VariableI_Name>(setA);            // 整數
engine.BuildBVs<VariableB_Name>(setA, setB);      // 二元
engine.BuildCVs<VariableX_Name>(lb, ub, setA);    // 自訂界限

// 建約束 / 目標（Pool）
engine.AddLHS(coef, new VariableX_Name { Set = s });
engine.AddRHS(value);
engine.CreateLessEqual("Name");                  // 或 CreateGreatEqual / CreateEqual
engine.CreateMaximize();                         // 或 CreateMinimize

// 求解與讀解
double obj = engine.GetObjectiveValue();
var sol = engine.GetSetVarValues<VariableX_Name>();   // 變數值
天條清單（人與 AI 都要守）

① 數值全進 Parameter.QTY，任何地方不得裸數字。
② Pool 不移項、不改號、不翻轉比較方向、不合併化簡、不四捨五入。
③ Parameter/Variable 只宣告 properties、用 object initializer；QTY 放最後。
④ 先 AddVariables 後 AddModel；BuildModel 內先 Objective 再 Constraints。
⑤ 先想清楚數學（過點①），再寫 code；正確性 gate 沒過不進 tuning。
